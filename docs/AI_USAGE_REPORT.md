## 一、项目 AI 使用概览

### 1.1 核心 AI 能力

本项目的核心是一个基于多 Agent 架构的智能任务拆解系统，使用大型语言模型（LLM）帮助用户将宏大的目标分解为可执行的小任务。

**AI 主要功能**:

| 功能模块 | 描述 | 使用频率 |
|---------|------|---------|
| 任务类型分析 | 自动识别用户目标属于哪一类任务 | 每次拆解必用 |
| 经验水平评估 | 根据用户自评和goal，给出更精准的水平判断 | 每次拆解必用 |
| 时间跨度判断 | 根据截止日期和每日可用时间，确定拆解层级 | 每次拆解必用 |
| 补充问题生成 | 生成高信息增益的后续问题，优化任务计划 | 每次拆解必用 |
| 任务拆解生成 | 生成月度→周度→日度的详细任务计划 | 每次拆解必用 |
| 快速任务拆解 | 为"快速上手"类目标生成节点框架和操作指南 | 快速模式专用 |

### 1.2 AI 模型配置

| 模型类型 | 模型名称 | 用途 | 调用次数/天 (预估) |
|---------|---------|------|-------------------|
| 快速模型 | `inclusionAI/Ling-flash-2.0` | Agent 1-3 分析 + QuickTask Agent A | 3×N + M |
| 思考模型 | `moonshotai/Kimi-K2-Thinking` | Agent 4-5 生成任务和问题 | 2×N |
| 快速模型 | `inclusionAI/Ling-flash-2.0` | QuickTask Agent B | 1×W×M |

**符号说明**:
- N = 标准任务拆解请求数
- M = 快速任务拆解请求数
- W = 平均每个快速任务的节点数

## 二、多 Agent 架构设计

### 2.1 标准任务拆解流程 (Agent 1-6)

项目采用**两阶段并行处理**的多 Agent 架构，最大程度利用 LLM 的并行能力：

```
用户提交表单
     ↓
┌─────────────────────────────────────────────┐
│  阶段1：分析 Agents (3个并行)                │
├─────────────────────────────────────────────┤
│  Agent 1 → 任务类型分析                      │
│  Agent 2 → 经验水平评估                      │
│  Agent 3 → 时间跨度判断                      │
└───────────────┬─────────────────────────────┘
                │ 汇总分析结果
                ↓
┌─────────────────────────────────────────────┐
│  阶段2：生成 Agents (2个并行)                │
├─────────────────────────────────────────────┤
│  Agent 5 → 任务拆解 (月度→周度→日度)          │
│  Agent 4 → 补充问题生成                       │
└───────────────┬─────────────────────────────┘
                │
                ↓
          返回结果给前端
```

**设计优势**:
- **减少总耗时**: 阶段1和阶段2内部并行，相比串行减少约 50% 等待时间
- **职责清晰**: 每个 Agent 专注单一任务，提示词更简单精准
- **容错性强**: 某个 Agent 失败可使用默认值兜底，不影响整体流程
- **易于扩展**: 新增 Agent 只需添加新的处理函数，不影响现有逻辑

### 2.2 快速任务拆解流程 (Agent A & B)

快速任务模式采用**双链路并行处理**：

```
用户输入想法
     ↓
┌─────────────────────────────────────────────┐
│  阶段1：并行执行 (2个 Agent)                  │
├─────────────────────────────────────────────┤
│  Agent A → 提取节点框架 (3-8个步骤)           │
│  搜索专业资料 → AI 内置知识库                 │
└───────────────┬─────────────────────────────┘
                │
                ↓
┌─────────────────────────────────────────────┐
│  阶段2：Agent B 生成操作指南                   │
├─────────────────────────────────────────────┤
│  为每个节点生成:                              │
│  - 具体步骤 (3-6步)                           │
│  - 完成标准 (2-4条)                           │
│  - 注意事项 (2-4条)                            │
└───────────────┬─────────────────────────────┘
                │
                ↓
          返回检测节点 + 操作指南
```

## 三、AI 模型使用详情

### 3.1 模型选择策略

#### 快速模型: `inclusionAI/Ling-flash-2.0`

**用于场景**:
- Agent 1: 任务类型分析
- Agent 2: 经验水平评估
- Agent 3: 时间跨度判断
- QuickTask Agent A: 节点框架提取
- QuickTask 资料搜索

**选择理由**:
- 响应速度快 (目标: <2秒)
- 简单分类任务准确率高
- 成本低，适合高频调用

**提示词特点**:
- 简单直接，要求单轮响应
- 明确的输出格式约束
- 低 temperature (0.3) 保证稳定性

**配置参数**:
```python
{
    "model": "inclusionAI/Ling-flash-2.0",
    "temperature": 0.3,
    "max_tokens": 8192,
    "timeout": 120.0  # 秒
}
```

#### 思考模型: `moonshotai/Kimi-K2-Thinking`

**用于场景**:
- Agent 4: 补充问题生成
- Agent 5: 任务拆解生成
- QuickTask Agent B: 操作指南生成

**选择理由**:
- 更强的推理能力
- 适合复杂任务拆解和多轮思考
- JSON 结构生成更稳定

**提示词特点**:
- 详细的系统提示词 (1000+ tokens)
- 明确的输出格式要求
- 较高的 temperature (0.7) 保证创造性

**配置参数**:
```python
{
    "model": "moonshotai/Kimi-K2-Thinking",
    "temperature": 0.7,
    "max_tokens": 16384,  # 思考模型需要更多 tokens
    "timeout": 600.0  # 10分钟超时
}
```

### 3.2 API 调用策略

#### 错误处理与重试机制

所有 AI 调用都配备**指数退避重试**策略：

```python
max_retries = 3
for attempt in range(max_retries):
    try:
        response = client.chat.completions.create(...)
        return response
    except (APIConnectionError, APIError) as e:
        wait_time = 2 ** attempt  # 1s, 2s, 4s
        time.sleep(wait_time)
    except RateLimitError as e:
        wait_time = 5  # 速率限制等待更久
        time.sleep(wait_time)
```

**重试场景**:
- 网络连接错误 (APIConnectionError)
- API 服务器错误 (APIError, status >= 500)
- 请求超时 (Timeout)
- 速率限制 (RateLimitError)

**不重试场景**:
- 认证错误 (401/403) - 配置问题，重试无用
- 参数错误 (400) - 请求格式错误，重试无用
- JSON 解析失败 - 内容问题，重试可能仍失败

#### 超时配置

| 模型类型 | 超时时间 | 说明 |
|---------|---------|------|
| 快速模型 | 120 秒 | 单次请求，足够处理简单任务 |
| 思考模型 | 600 秒 (10分钟) | 复杂推理需要更长时间 |

#### SSL 配置

**特殊处理**: Windows 环境下禁用 SSL 证书验证

```python
self.client = OpenAI(
    api_key=api_key,
    base_url=base_url,
    http_client=httpx.Client(
        verify=False,  # 临时方案
        timeout=120.0
    )
)
```

**原因**: Windows 上的 httpx 默认使用系统证书库，可能缺少中间证书导致连接失败。

## 四、提示词工程 (Prompt Engineering)

### 4.1 Agent 1-3: 分析类提示词

**设计原则**:
- 简洁明确，避免歧义
- 提供输出格式示例
- 限制输出长度 (≤100 字符)

**示例 - Agent 1 任务类型分析**:
```
分析以下目标属于哪种任务类型，只返回类型名称和简短描述（50字以内）。

目标：{goal}

常见任务类型：
- 技能学习类：学习编程、学习语言、学习乐器等
- 项目开发类：开发网站、开发APP、写毕业论文等
- 健康健身类：减肥、增肌、跑步训练等
- ...

返回格式：类型名称 - 简短描述
例如：技能学习类 - 网页开发
```

**技巧**:
- 给出明确的 few-shot 示例
- 使用 "只返回..." 约束输出
- 提供常见分类列表，减少模型思考范围

### 4.2 Agent 4: 补充问题生成提示词

**设计原则**:
- 角色设定清晰 ("补充问题生成器")
- 明确职责边界 ("不负责生成计划")
- 三维度覆盖: 个人偏好、个人基础、任务优先级
- 高信息增益优先

**关键约束**:
```
输出规则：
1. 高信息增益：优先问若回答会显著改变任务结构或排程的因素
2. 可执行性相关：围绕时间/范围/质量标准/资源/约束/依赖/风险
3. 避免重复：不要问用户已填写过的问题
4. 可选语气：用户可以跳过，不用强制性语言
5. 保护隐私：不索要不必要的敏感信息
```

**类别推断**:
通过关键词自动为问题分配 `category` 字段：
- `preference`: 学习、方式、偏好、喜欢、倾向...
- `foundation`: 经验、基础、能力、技能、资源...
- `priority`: 优先级、质量、标准、时间、分配...

### 4.3 Agent 5: 任务拆解提示词

**设计原则**:
- 系统提示词详细定义角色和输出格式
- 用户提示词包含所有约束条件
- 严格的 JSON 格式要求

**系统提示词结构**:
```
你是 Agent 6 - 专业任务拆解器

## 你的输出格式
（详细JSON schema，包含 monthly/weekly/daily 结构）

## 拆解原则
- 月度任务：描述整体目标和最终产出
- 周度任务：明确本周要达成什么，用"产出："开头
- 日度任务：每天1小时内能完成的具体操作

## 重要规则
1. 每日任务必须可执行
2. 产出导向
3. 时间约束：假设每天只有1小时
4. 渐进式：从简单到复杂
5. 每周最后一天设为"机动"日
```

**用户提示词包含**:
- 目标描述
- 时间约束 (每天可用时间、总周期)
- AI 分析结果 (任务类型、经验水平、时间跨度)
- 日期格式示例

### 4.4 QuickTask 提示词

**Agent A - 节点提取**:
```
你是"任务节点提取专家"
要求：
1. 3-8个步骤
2. 步骤之间有先后顺序
3. 每个步骤有预估时间（15分钟-2小时）
4. 标注依赖关系
5. 步骤名称简洁明确
```

**Agent B - 操作指南**:
```
你是"任务执行专家"
当前节点：{node_name}

请为这个节点生成具体的操作指南:
- description: 节点概述
- steps: 3-6个具体可执行的步骤
- completion_criteria: 2-4条完成标准
- pain_points: 2-4条常见错误提醒
```

```
┌─────────────────────────────────────────────────────────────┐
│  步骤1: 用户提交表单                                        │
├─────────────────────────────────────────────────────────────┤

│    "experience": "beginner",                               │

│    ...                                                     │
│  }                                                          │
└───────────────────────────┬─────────────────────────────────┘
                            │

│  ThreadPoolExecutor(max_workers=3)                         │
│  - _agent_task_type() → "技能学习类 - 网页开发"            │
│  - _agent_experience() → "零基础 - 需要从基础概念开始"     │

                            │
┌───────────────────────────▼─────────────────────────────────┐
│  步骤3: 汇总分析结果                                         │

│  }                                                          │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  步骤4: 并行调用 Agent 4-5 (生成阶段)                        │
├─────────────────────────────────────────────────────────────┤
│  ThreadPoolExecutor(max_workers=2)                         │
│  - _agent_breakdown() → {monthly, weekly, daily}          │
│  - _agent_questions() → [q1, q2, q3]                      │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  步骤5: 组装并返回结果                                       │
├─────────────────────────────────────────────────────────────┤
│  {                                                          │
│    "project_id": "uuid",                                   │
│    "analysis": {...},                                      │
│    "tasks": {monthly, weekly, daily},                      │
│    "follow_up_questions": [...]                            │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 重新生成流程

当用户回答补充问题后，触发重新生成：

```
用户提交答案
     ↓
后端收到 POST /api/projects/{id}/answers
     ↓
读取之前的 tasks 和 analysis
     ↓
构建优化提示词（包含原始任务摘要 + 用户答案）
     ↓
调用 Agent 5 重新生成任务
     ↓
调用 Agent 4 生成新的补充问题（避免重复）
     ↓
返回新的 tasks + follow_up_questions
```

**关键点**:
- 保留之前的任务结构作为参考
- 新提示词包含 "原始任务计划摘要" 部分
- Agent 4 接收 `previous_questions` 参数，避免重复提问

## 七、性能与成本分析

### 7.1 API 调用次数估算

**单次标准任务拆解**:
```
阶段1 (分析):
  Agent 1:  1 次调用 × N
  Agent 2:  1 次调用 × N
  Agent 3:  1 次调用 × N
小计: 3N 次

阶段2 (生成):
  Agent 5:  1 次调用 × N  (任务拆解，~8000 tokens)
  Agent 4:  1 次调用 × N  (补充问题，~2000 tokens)
小计: 2N 次

总计: 5N 次 API 调用
```

**单次快速任务拆解**:
```
阶段1:
  Agent A:    1 次调用 × M  (~4000 tokens)
  资料搜索:   1 次调用 × M  (~2000 tokens)
小计: 2M 次

阶段2:
  Agent B:    W 次调用 × M  (W=节点数，~2000 tokens/次)
小计: W×M 次

总计: (2+W)M 次 API 调用 (W≈5，约 7M 次)
```

### 7.2 Token 消耗估算

基于典型响应长度:

| Agent | 输入 tokens | 输出 tokens | 总计/次 |
|-------|------------|------------|---------|
| Agent 1-3 | ~500 | ~50 | 550 |
| Agent 4 | ~800 | ~300 | 1100 |
| Agent 5 | ~1000 | ~8000 | 9000 |
| QuickTask A | ~300 | ~400 | 700 |
| QuickTask B (每节点) | ~600 | ~500 | 1100 |

**单次标准拆解总 token**:
```
分析阶段: 3 × 550 = 1650 tokens
生成阶段: 9000 + 1100 = 10100 tokens
总计: ~11750 tokens
```

### 7.3 成本估算 (硅基流动)

基于公开价格 (2025-01):

| 模型 | 输入价格 (每1M tokens) | 输出价格 (每1M tokens) |
|------|----------------------|----------------------|
| Ling-flash-2.0 | ~¥0.5 | ~¥1.0 |
| Kimi-K2-Thinking | ~¥5.0 | ~¥10.0 |

**单次标准拆解成本**:
```
Agent 1-3 (Ling-flash):
  输入: 1650 × 0.5/1e6 = ¥0.0008
  输出: 150 × 1.0/1e6 = ¥0.00015
  小计: ~¥0.001

Agent 5 (Kimi-K2, 输出占大头):
  输入: 1000 × 5.0/1e6 = ¥0.005
  输出: 8000 × 10.0/1e6 = ¥0.08
  小计: ~¥0.085

Agent 4 (Kimi-K2):
  输入: 800 × 5.0/1e6 = ¥0.004
  输出: 300 × 10.0/1e6 = ¥0.003
  小计: ~¥0.007

总计: ~¥0.093/次 (约 9 分钱)
```

**1000次拆解成本**: ¥93 (仅模型调用)

> **注意**: 以上为估算值，实际成本取决于实际 token 消耗和 API 价格变动。

## 八、已知问题与优化建议

### 8.1 当前问题

#### 问题 1: JSON 解析失败处理不完善

**现象**: Agent 5 返回截断的 JSON 时，虽然已有修复逻辑，但仍可能失败。

**位置**: `backend/services/ai_service.py:842-906`

**建议**:
- 增加更严格的结构验证 (JSON Schema)
- 失败时自动重试一次（调整 prompt）
- 记录失败案例，用于后续模型微调

#### 问题 2: SSL 验证被禁用

**现象**: Windows 环境下 `verify=False` 存在安全风险。

**位置**: `backend/services/ai_service.py:32`

**建议**:
- 查找系统证书缺失的根本原因
- 使用 `certifi` 包提供证书
- 生产环境必须启用 SSL 验证

#### 问题 3: 补充问题可能重复

**现象**: 即使有 `previous_questions` 参数，Agent 4 仍可能生成相似问题。

**位置**: `backend/services/ai_service.py:259` 和 `:381-422`

**建议**:
- 改进 `_ensure_category_fields` 的逻辑，更严格去重
- 在 prompt 中增加更强的去重约束
- 记录问答历史，避免跨会话重复

#### 问题 4: 缺少使用量统计

**现象**: 无法知道每天/每月 AI API 调用次数和成本。

**建议**:
- 增加 API 调用计数器 (Redis/数据库)
- 记录每次调用的 token 消耗
- 添加管理后台查看使用情况

#### 问题 5: 超时时间固定

**现象**: 所有思考模型使用 600 秒超时，可能不够或过长。

**建议**:
- 根据任务复杂度动态调整超时
- 添加流式响应，逐步返回结果
- 实现取消机制，允许用户中断

### 8.2 短期优化 (1-2 周)

1. **增加缓存层**
   - 缓存相同目标的拆解结果 (Redis/Memory)
   - 缓存补充问题和答案的关联

2. **优化 prompt**
   - 压缩系统提示词，减少 token 消耗
   - 针对常见目标类型预设模板

3. **添加监控**

   - 提升用户体验

3. **用户反馈机制**

   - 用于优化 prompt 和模型选择

### 8.4 长期规划 (3-6 月)

## 九、AI 安全与伦理

### 9.1 内容安全

**当前状态**:
- 依赖硅基流动平台的内容过滤机制
- 未实现自定义内容审核

**建议**:
- 添加敏感词过滤 (用户输入和目标描述)
- 记录异常请求，人工审核

### 9.2 数据隐私

**当前状态**:
- 用户数据直接发送至第三方 AI API
- 未在本地进行数据脱敏

**建议**:
- 明确告知用户数据将发送至 AI 服务商
- 提供"禁用 AI"的本地模式 (使用预设模板)
- 定期清理历史数据

### 9.3 输出可靠性

**当前风险**:
- AI 生成的任务可能不准确或不完整
- 用户依赖错误任务可能导致失败

**缓解措施**:
- 提供"重新生成"功能
- 允许用户手动编辑任务
- 添加免责声明
